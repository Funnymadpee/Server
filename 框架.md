==============================================================================
客户端包： 长度 + 协议id + 数据

gate包： gateid connid 消息类型 协议id  数据长度 数据


router: 根据协议id进行路由

各服务器通过ServerConnection进行与其他服务器发起主动连接

全局服和场景服本来想用Trinitycore的设计思路 采用多线程处理业务,uid绑定线程标识,想想还是算了，处理业务单线程安全点
=======================================================================
[GateServer]
1. 管理连接session
2. 保活/心跳
3. 粘包拆包

Gate 纯异步透传，Router 管全局映射，因为无状态支持集群


[GateRouterServer]
1. 解析协议
2. 绑定uid
3. 后续路由

Router 只认 GlobalKey = GateID + ConnID
客户端A → Gate1(gateId=1) connId=1

网络线程收消息  --> 消息队列 -->  逻辑线程
1 个 网络线程（高吞吐）
N 个逻辑线程（高并发）

几个关系映射：
1. gateid + connid ===> uid   向前
2. 协议号 ===>  服务器类型  提供大致转发方向
3. 玩家id  ====>  玩家与业务Server的绑定信息  ===>  serverid  精确方向
4. 在场景服下 serverid ---> scenceid


对于多个场景服务器  router要记录玩家与场景服的关系
第一次进入选择默认场景服 场景服将玩家位置信息同步给全局游戏逻辑服  全局游戏逻辑服再同步给router


[GlobalLogicServer]
全局游戏逻辑服
1. 解析协议, 调用相关接口
2. 处理场景服的调度，玩家等数据
      玩家登录、身份验证
      在线状态管理
      跨场景、跨服调度
      禁言、踢人、全服公告
      服务发现
3. 同步相关状态数据给router

网络线程收消息  --> 队列 --> 主线程消费
开几个场景服，构成的场景服表 同步给router 以方便router直连场景服

全局逻辑服 → 发消息给中间件 → 转发给对应微服务
c++ ->  nats ->  go


[ScenceServer]
场景服：玩家玩的地方
1. 游戏相关的功能  跑图 AOI 战斗 场景逻辑 玩家逻辑 ai
主线程接受网络数据包进行分发处理
然后世界更新
      -->场景更新
            -->场景内对象更新
                  -->对象身上数据更新
